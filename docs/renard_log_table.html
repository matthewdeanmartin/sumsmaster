Renard Tables: Mental Multiplication Without Interpolation

Old mental-arithmetic books teach a beautiful trick: memorize a small table of base-10 logarithms, and you can multiply, divide, and take powers and roots of any numbers in your head, to within a few percent. The catch is that the classic version requires interpolating between table entries — a fiddly mental division most people won't do. This page explains a table that needs no interpolation at all: the Renard series, also called preferred numbers.

The log method in 20 seconds

Logarithms turn multiplication into addition: log(a·b) = log a + log b. So to multiply two numbers:

  1. look up the log of each number,
  2. add the logs (easy),
  3. look up which number has that log (the “antilog”).

Division is the same with subtraction. Square roots: halve the log. Powers: multiply the log. One table, four-plus operations. Powers of ten ride along for free: log 4870 = log 4.87 + 3, so you only ever need logs of numbers between 1 and 10.

The problem with the classic table

The book table stores log 1 through log 10 (log 2 ≈ 0.30, log 3 ≈ 0.48, …). But what is log 4.87? It's between log 4 = 0.60 and log 5 = 0.70, about 87% of the way — you must interpolate. Skip that step and just snap 4.87 to its nearest neighbor, and accuracy collapses: in simulation, the median multiplication error jumps from 2.6% to 14% — worse than doing no log trick at all. The gaps between entries are widest, in log terms, exactly where most real-world numbers live (1.0–2.0).

Renard's idea: space the table evenly in log

In 1877, French military engineer Colonel Charles Renard was asked to rationalize the sizes of balloon mooring cables. He cut 425 sizes down to 17 by making each size a constant ratio larger than the last — evenly spaced in log space. The scheme became ISO 3, the “preferred numbers” used today for resistor values, decibels, camera stops, and 1–2–5 currency denominations.

The 10-step series, R10, picks the round-looking number closest to each power 10k/10:

log (memorized: count by 0.1) 0.00.10.20.30.4 0.50.60.70.80.91.0
number (memorized) 11.251.622.5 3.15456.3810
true 10k/10 11.2591.5851.9952.512 3.1623.9815.0126.3107.94310

Notice what happened to the memorization burden. The logs are now just “count by 0.1” — a rule, not facts. All you memorize is the number row, and most of it is structural too (see the doubling rule below). Total: about 7 digits of genuine memorization, versus 16 for the classic table.

And because the entries are evenly spaced in log, snapping to the nearest entry is all you ever do — every snap costs at most half a step (±0.05 in log, about ±12%), no matter where the number falls.

Worked examples (R10, no interpolation)

Multiply: 1.6 × 2.5
1.6 → 0.2 2.5 → 0.4 0.2 + 0.4 = 0.6 → 4

Answer: 4. True value: 4.0 — exact.

Multiply: 487 × 2,310
487 = 4.87 × 10² 4.87 snaps to 5 → 0.7 + 2 = 2.7 2,310 = 2.31 × 10³ 2.31 snaps to 2.5 → 0.4 + 3 = 3.4 2.7 + 3.4 = 6.1 → fractional part 0.1 → 1.25 answer ≈ 1.25 × 10⁶

True value: 1,124,970 — estimate is 11% high. R10 is a rough tool; the R20 version below fixes this.

Divide: 6,400 ÷ 18
6,400 = 6.4 × 10³ 6.4 snaps to 6.3 → 0.8 + 3 = 3.8 18 = 1.8 × 10¹ 1.8 → between 1.6 (0.2) and 2 (0.3) → call it 0.25 + 1 3.8 − 1.25 = 2.55 → 0.55 is between 0.5 (3.15) and 0.6 (4) → ≈ 3.55 answer ≈ 355

True value: 355.6 — within 0.2%. (With R20 the in-between values 0.25 → 1.8 and 0.55 → 3.55 are actual table entries, no judgment needed.)

Square root: √5000
5,000 = 5 × 10³ → log ≈ 0.7 + 3 = 3.7 halve: 3.7 ÷ 2 = 1.85 → fractional 0.85, between 6.3 (0.8) and 8 (0.9) → ≈ 7.1 answer ≈ 7.1 × 10¹ = 71

True value: 70.71 — within 0.5%. Try doing that by rounding-and-multiplying.

Power: 2¹⁰
log 2 = 0.3 ×10 → 3.0 → 10³ answer ≈ 1,000

True value: 1,024 — 2.3% low. (This is why log 2 ≈ 0.3 in the first place: 2¹⁰ ≈ 10³.)

Caution: powers multiply the table's stored error by the exponent. 1.4¹⁰ via the table gives 0.15 × 10 = 1.5 → 31.5, but the truth is 28.9 (9% off) because log 1.4 = 0.146, not 0.15.

The R20 upgrade: half-steps

Twenty steps per decade (each step 0.05 in log) doubles the precision. Highlighted cells are the only new “seeds” you memorize — the rest of the row is the R10 table or generated by the doubling rule.

log 0.000.050.100.15 0.200.250.300.35 0.400.450.50
number 11.121.251.4 1.61.822.24 2.52.83.15
log 0.550.600.650.700.75 0.800.850.900.951.00
number 3.5544.555.6 6.37.18910

Redo the multiplication that R10 fumbled:

Multiply: 487 × 2,310 (R20)
4.87 snaps to 5 → 0.70 + 2 2.31 snaps to 2.24 → 0.35 + 3 2.70 + 3.35 = 6.05 → 0.05 → 1.12 answer ≈ 1.12 × 10⁶

True value: 1,124,970 — within 0.4%.

How to memorize it: the doubling rule

Because ×2 is +0.30 in log, doubling a number moves it exactly 3 slots along R10 (6 slots along R20):

1 → 2 → 4 → 8   |   1.25 → 2.5 → 5 → 10   |   1.6 → 3.15 → 6.3

So R10 is generated by doubling three seeds: 1, 1.25, 1.6. R20 adds three more: 1.12, 1.4, 1.8 (giving 2.24, 2.8, 3.55, 4.5, 5.6, 7.1, 9 by doubling). Six small numbers and one rule reconstruct the whole table.

You may already know this table. Each R10 step is exactly 1 decibel of power ratio — “3 dB = double” is the doubling rule. The entry at 0.5 is √10 ≈ 3.16. Resistor E-series, camera f-stops, and 1–2–5 coin denominations are all Renard-flavored. Engineers who “think in dB” are running this exact algorithm.

Reading the table: snapping on R20 roughly matches the classic interpolated method while skipping its hardest mental step, and beats it per memorized digit. If you later learn to interpolate (“2.31 is a third of the way from 2.24 to 2.5”), the same R20 table becomes the best small table tested. And unlike round-and-multiply, every row of this table also does division, roots, and powers.

When not to use it

  • Numbers near 1 (1.05¹⁵ compound interest): the table can't resolve them; use ln(1+x) ≈ x — i.e. log(1+x) ≈ 0.43·x.
  • When you need an exact answer — this is an estimation tool; pair it with a last-digit or casting-out-nines check if exactness matters.
  • Addition and subtraction — logs don't help; just round and add.
